## 摘要
大型语言模型（LLM）智能体越来越多地应用于工程设计任务，但现有评估框架未能充分处理结合模拟、检索和制造准备的多智能体系统。我们引入了一个包含三个评估维度的基准套件：（1）一个工作流基准，包含七种针对不同认知需求的提示风格——包括直接工具使用、语义消歧、条件分支和工作记忆任务；（2）一个检索增强生成（RAG）基准，通过门控评分隔离检索对参数选择的贡献；（3）一个高性能计算（HPC）基准，评估在SLURM集群上的端到端机器学习训练编排。与基准一同，我们提出了EngiAI，一个基于LangGraph构建的多智能体系统（MAS）参考实现，通过监督架构协调七个专业智能体，统一了拓扑优化、文档检索、HPC作业编排和3D打印机控制。在四个LLM后端和两个EngiBench问题上，专有模型在Beams2D上达到了96-97%的平均任务完成率，而开源4B参数模型达到了55-78%，且代际改进明显。条件分支最具挑战性，在Photonics2D上条件风格的任务完成率降至20-53%。RAG门控确认了接近完美的检索增强分数（≈1.0），而无检索时接近零，验证了评估设计。在HPC编排上，一个模型在100%的运行中完成了所有流水线步骤，而另一个模型降至50%，揭示了多步骤指令遵循在长时间运行工作流中会退化。

## Abstract
Large Language Model (LLM) agents are increasingly applied to engineering design tasks, yet existing evaluation frameworks do not adequately address multi-agent systems that combine simulation, retrieval, and manufacturing preparation. We introduce a benchmark suite with three evaluation dimensions: (1) a workflow benchmark with seven prompt styles targeting distinct cognitive demands-including direct tool use, semantic disambiguation, conditional branching, and working-memory tasks; (2) a Retrieval-Augmented Generation (RAG) benchmark with gated scoring isolating retrieval contributions to parameter selection; and (3) an High Performance Computing (HPC) benchmark evaluating end-to-end ML training orchestration on a SLURM cluster. Alongside the benchmark we present EngiAI, a Multi-Agent System (MAS) reference implementation built on LangGraph that operationalizes the benchmark by coordinating seven specialized agents through a supervisor architecture, unifying topology optimization, document retrieval, HPC job orchestration, and 3D printer control. Across four LLM backends and two EngiBench problems, proprietary models achieve 96-97% average task completion on Beams2D, while open-source 4B-parameter models reach 55-78%, with clear generational improvement. Conditional branching proves most challenging, with task completion dropping to 20-53% for the conditional style on Photonics2D. RAG gating confirms near-perfect retrieval-augmented scores ($\approx 1.0$) versus near-zero without retrieval, validating the evaluation design. On HPC orchestration, one model completes all pipeline steps in 100% of runs while another drops to 50%, revealing that multi-step instruction following degrades over long-running workflows.

- **核心问题**：现有LLM评估框架（如AgentBench、ToolSandbox等）主要面向单智能体或通用工具调用任务，缺乏针对**工程设计**中多智能体协作场景（包括模拟、检索、制造准备）的标准化评估基准。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：构建一个监督式多智能体系统（MAS），通过中央监督智能体路由用户查询到七个专业智能体，每个专业智能体拥有小而专注的工具集，避免工具选择退化。同时设计阶梯式基准套件，包含多种认知需求（直接工具使用、语义消歧、条件分支、派生参数计算、工作记忆等）。
- **系统架构**：
  - **监督智能体**：负责意图分类、路由和终止检测（防止无限循环）。
  - **七个专业智能体**：工程智能体（通过EngiBench/EngiOpt进行拓扑优化、仿真、STL导出）、RAG智能体、ArXiv智能体（两者基于MMORE框架实现检索）、搜索智能体（Tavily API）、HPC智能体（管理SLURM集群作业）、CLI智能体（执行本地shell命令）、Prusa智能体（控制3D打印机，通过MCP协议）。
- **基准套件设计**：
  - **工作流基准**：七种提示风格：FULL（直接给全参数）、NATURAL（仅定性描述，需歧义检测）、W-RAND（随机STL参数）、W-DERIVED（需算术推导）、W-DISTRACT（竞争参数值）、W-COND（基于模拟结果的条件分支）、W-MULTI（两次不同导出）。每种风格施加不同的认知负载。

- **评分方法**：
  - 工作流：综合得分 = 0.65×设计质量（IoU、像素精度、目标匹配、约束匹配、2D连通性、3D水密性）+ 0.20×工具效率 + 0.15×任务完成率。
  - RAG：加权参数准确度仅当调用了搜索工具才计入，否则为0。
  - HPC：0.70×步骤完成率 + 0.15×配置正确性 + 0.15×评估指标提取。

## 6. 论文的主要结论与发现

- **RQ1（工作流性能）**：专有模型在Beams2D上平均任务完成率96-97%，开源Qwen3.5-4B达78%（Qwen3-4B为55%），存在明显代际进步。条件分支（W-COND）最难，在Photonics2D上最佳模型仅53%，且所有失败均为分支反转（选相反分支），表明模型能解析条件但比较方向出错。
- **RQ2（模型鲁棒性）**：GPT-5-mini和Gemini-3-Flash表现一致；开源4B模型代际改进可部分弥补规模不足。
- **RQ3（工具效率）**：多余工具调用会降低综合得分（从0.70降至0.63），而设计质量保持不变（~0.5），说明效率影响评分但工程输出鲁棒。Qwen3.5-4B在多数风格上达到最优工具效率。
- **RQ4（RAG贡献）**：RAG-on得分接近1.0，RAG-off为0（因门控机制），Empty RAG显著退化，证明参数选择依赖检索内容而非先验知识。对于后训练截止点论文（arXiv:2505.02438）的参数，无检索无法获得。
- **RQ5（HPC编排）**：Gemini-3-Flash在100%运行中完成全流水线；GPT-5-mini在显式指令下达70%，自然语言下仅50%，揭示长流程指令遵循退化。失败模式主要是跳过最后评估步骤。

## 7. 优点

- **全面性**：首次将工作流执行、RAG和HPC编排三大维度统一在一个基准套件中，覆盖多智能体工程设计全链条。
- **精细认知设计**：七种提示风格系统性地测试不同认知负载（数值精度、语义消歧、条件推理、工作记忆等），比现有工具调用基准更深入。
- **门控RAG评分**：干净地隔离检索贡献，防止模型通过参数记忆获得虚假分数，设计巧妙。
- **开源参考实现**：EngiAI基于LangGraph提供可复现的模块化架构，便于后续扩展和复现。
- **实验多样性**：涵盖专有和开源模型（含不同代际4B模型），两个不同物理领域的问题，统计量报告均值和标准差。

## 8. 不足与局限

- **问题覆盖面有限**：仅测试2个EngiBench问题（Beams2D和Photonics2D），未涵盖其他物理领域。

- **HPC评估不完整**：仅使用cGAN算法，扩散模型结果放在附录；未测试开源4B模型在HPC上的表现。
- **未进行用户研究**：未邀请真实工程师进行交互式评估，缺少人机协作场景。
